What it is

runshot drives a fresh-account onboarding walkthrough of any web app in a mobile viewport and records the whole thing — so a human can see exactly what a new user sees.

It captures a video, a screenshot of every screen (including zero-states), and any emails the flow triggers. One flow definition produces two outputs: a human-reviewable QA artifact and a deterministic CI gate.

The engine is config-driven, so the same procedure works for every app — the per-app specifics live in that app's runshot/skills.config.json. Nothing about the target app is hard-coded.

One run, two outputs

record mode

A QA artifact

A browsable gallery — phone-framed screens laid out in flow order, the session video, captured emails, and link-preview mockups. Something you want to share.

assert mode

A CI gate

The same capture, but it exits non-zero on failure. Drop it into CI and every PR proves the onboarding flow still renders, end to end.

Two skills

walkthrough

Drives the onboarding flow; records video + per-screen screenshots, captures emails, and writes manifest.json + summary.json.

flow-doc

Turns a run's screens into an editable flow diagram — one frame per screen, in flow order — as flow.drawio (open in draw.io) or via the Figma MCP write path.

Quick start

# one-time: install Chromium
npx runshot setup
# capture a run from your config
npx runshot record --config ./runshot/skills.config.json
# browse captured runs at :8080
npx runshot serve

runshot assert is the same capture but exits non-zero on failure — use it as a CI gate.

Browse anywhere

The gallery hub serves a phone-framed view of every project's runs. It mounts cleanly at the root, under a sub-path (BASE_PATH=/runshot), or on its own subdomain — every internal link, asset, and API call follows the base path, so it works behind a reverse proxy without escaping its prefix.
